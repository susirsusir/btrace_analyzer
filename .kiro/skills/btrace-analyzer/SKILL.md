## Overview

This skill analyzes BTrace/xTrace format Android performance sampling traces. These are custom binary trace files generated by `com.xmhaibao.base.monitor.BTraceMonitor` or similar Android performance monitoring SDKs. They are NOT standard Perfetto/systrace format and cannot be loaded in Perfetto UI directly.

## Input Requirements

Two files are needed:
- **Trace file (sampling)**: Binary file containing sampled stack frames with timing data
- **Mapping file (sampling-mapping)**: Binary file mapping memory addresses to Java method signatures

## Analysis Workflow

### Step 1: Download Files

Download both the trace file and mapping file to `/tmp/`:

```bash
curl -L -o /tmp/trace_sampling "<trace_url>"
curl -L -o /tmp/trace_sampling_mapping "<mapping_url>"
```

### Step 2: Verify File Format

Check the binary header to confirm BTrace format:

```bash
xxd /tmp/trace_sampling | head -5
xxd /tmp/trace_sampling_mapping | head -5
```

**BTrace trace file signature:**
- Magic number: `04 03 02 01` (first 4 bytes)
- Contains JSON metadata like `{"processId":XXXXX}` near the header
- Version field at offset 8 (typically version 5)

**Mapping file signature:**
- Starts with 16-byte header
- Contains Java method signatures as readable strings (e.g., `void com.xxx.ClassName.method()`)

### Step 3: Parse Mapping File

The mapping file has this binary structure:

```
Header (16 bytes):
  - 8 bytes: reserved (typically 0)
  - 4 bytes: version (uint32 LE, typically 1)
  - 4 bytes: method count (uint32 LE)

Records (repeated):
  - 8 bytes: method address (uint64 LE) - memory address of the compiled method
  - 2 bytes: string length (uint16 LE)
  - N bytes: method signature string (UTF-8)
```

Parse with Python and save as JSON for later use:

```python
import struct, json

with open('/tmp/trace_sampling_mapping', 'rb') as f:
    data = f.read()

pos = 16  # skip header
records = {}
while pos + 10 <= len(data):
    addr = struct.unpack_from('<Q', data, pos)[0]
    str_len = struct.unpack_from('<H', data, pos + 8)[0]
    pos += 10
    if str_len > 0 and str_len < 2000 and pos + str_len <= len(data):
        method_name = data[pos:pos+str_len].decode('utf-8', errors='replace')
        records[addr] = method_name
        pos += str_len
    else:
        break

# Save as JSON (hex keys)
mapping_json = {hex(k): v for k, v in records.items()}
with open('/tmp/trace_mapping.json', 'w') as f:
    json.dump(mapping_json, f)
```

### Step 4: Parse Trace File

The trace file has this binary structure:

```
File Header:
  - 4 bytes: magic (0x01020304, LE)
  - 4 bytes: reserved
  - 4 bytes: version (uint32 LE)
  - 8 bytes: timestamp
  - 4 bytes: sampling interval in microseconds (e.g., 25000 = 25ms)
  - 4 bytes: JSON metadata length (uint32 LE)
  - N bytes: JSON metadata string (e.g., {"processId":11550})

Sample Records (repeated until EOF):
  - 2 bytes: marker (various values: 0x0002, 0x0007, 0x0008, 0x0009, 0x000A, 0x000B, 0x0011, 0x0017)
  - 66 bytes: timing and thread data (timestamps, thread IDs, etc.)
  - 4 bytes: stack depth (uint32 LE) at offset +68 from record start
  - 4 bytes: stack depth duplicate (uint32 LE) at offset +72
  - stack_depth * 8 bytes: frame addresses (uint64 LE each)

Total fixed header per record = 76 bytes before frame addresses.
```

### Step 5: Full Analysis Script

Create and run this Python script to produce the complete analysis:

```python
import struct, json
from collections import Counter

# Load files
with open('/tmp/trace_sampling', 'rb') as f:
    data = f.read()
with open('/tmp/trace_mapping.json', 'r') as f:
    mapping = json.load(f)

addr_map = {int(k, 16): v for k, v in mapping.items()}

# Parse header
json_len = struct.unpack_from('<I', data, 24)[0]
rec_start = 28 + json_len
HEADER_SIZE = 76

pos = rec_start
top_methods = Counter()
all_methods = Counter()
stack_counter = Counter()
sample_count = 0
bad = 0

while pos + HEADER_SIZE <= len(data):
    depth = struct.unpack_from('<I', data, pos + 68)[0]
    if depth <= 0 or depth >= 500:
        pos += 1
        bad += 1
        if bad > 200:
            break
        continue
    frame_end = pos + HEADER_SIZE + depth * 8
    if frame_end > len(data):
        break
    frames = []
    for i in range(depth):
        addr = struct.unpack_from('<Q', data, pos + HEADER_SIZE + i * 8)[0]
        name = addr_map.get(addr, f'native:0x{addr:x}')
        frames.append(name)
    if frames:
        top_methods[frames[0]] += 1
        seen = set()
        for f in frames:
            if f not in seen:
                all_methods[f] += 1
                seen.add(f)
        stack_key = '|'.join(frames[:5])
        stack_counter[stack_key] += 1
    sample_count += 1
    pos = frame_end

def short_name(m):
    return m.split('(')[0] if '(' in m else m

print(f'Total samples: {sample_count}, errors: {bad}')

print(f'\n=== TOP 30 SELF TIME (top of stack = direct CPU consumer) ===')
for method, count in top_methods.most_common(30):
    pct = count / sample_count * 100
    print(f'  {pct:6.2f}% ({count:6d}) {short_name(method)}')

print(f'\n=== TOP 30 INCLUSIVE TIME (anywhere in call stack) ===')
for method, count in all_methods.most_common(30):
    pct = count / sample_count * 100
    print(f'  {pct:6.2f}% ({count:6d}) {short_name(method)}')

print(f'\n=== TOP 15 HOT CALL STACKS (top 5 frames) ===')
for stack, count in stack_counter.most_common(15):
    pct = count / sample_count * 100
    parts = stack.split('|')
    print(f'\n  {pct:5.2f}% ({count}x):')
    for p in parts:
        print(f'    {short_name(p)}')
```

### Step 6: Interpret Results

When analyzing the output, focus on:

1. **Self Time (top of stack)**: Methods directly consuming CPU. Ignore `<runtime method>.<runtime internal callee-save reference registers method>` as it's ART runtime overhead.

2. **Inclusive Time**: Methods that appear anywhere in the call stack. High inclusive time means the method (or its callees) is responsible for significant CPU usage.

3. **Hot Call Stacks**: The most frequently sampled call paths. These reveal the actual execution flow causing performance issues.

**Common patterns to look for:**
- **Main thread blocking**: High samples in `MessageQueue.next` (idle) vs actual work on main thread
- **JSON serialization**: `JSONObject.put`, Gson `ReflectiveTypeAdapterFactory` - consider code-gen alternatives
- **Reflection overhead**: `java.lang.reflect.*` methods in hot paths
- **I/O on wrong thread**: File/network operations on main thread
- **Vendor SDK overhead**: OEM-specific code (vivo, huawei, etc.) consuming unexpected CPU
- **Database contention**: Room/SQLite operations with high frequency
- **Animation overhead**: Choreographer/VSync callbacks with heavy work
- **Logging overhead**: File-based logging in hot paths

## Notes

- The sampling interval is typically 25ms (25000μs), meaning each sample represents ~25ms of execution time
- Marker values (0x0002-0x0017) may represent different thread types or sampling contexts
- Addresses in the trace that don't match the mapping file are typically native/system methods
- This format is specific to BTrace/xTrace monitoring SDKs used in some Chinese Android apps
